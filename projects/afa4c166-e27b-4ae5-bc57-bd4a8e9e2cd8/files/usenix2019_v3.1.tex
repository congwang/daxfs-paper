\begin{abstract}
%-------------------------------------------------------------------------------
CXL (Compute Express Link) enables multiple hosts to share byte-addressable
memory with hardware cache coherence, creating disaggregated memory pools
accessible via direct load/store operations.  However, no existing filesystem
can exploit CXL shared memory for a multi-host writable namespace:
per-host filesystems (tmpfs, ext4-dax, NOVA) duplicate state across hosts,
while distributed filesystems impose network protocol overhead on what is
fundamentally a shared-memory problem.

We present \daxfs, a Linux filesystem designed for CXL shared memory.
\daxfs maps directly onto byte-addressable DAX memory and uses
\texttt{cmpxchg} atomic operations, which CXL makes coherent across host
boundaries, as its sole coordination primitive.  A CAS-based hash overlay
enables lock-free concurrent writes from multiple hosts without any
centralized coordinator.  A shared page cache in DAX memory provides
cooperative demand-paged caching that is automatically visible to all
participants.  The flat on-disk format enables bounded validation of untrusted
images with no pointer chasing.

On a 48-core Intel Xeon Gold 5418Y with Optane PMem, \daxfs achieves
$4.1\times$ higher sequential read throughput than ext4-dax and
$22\times$ higher than NOVA, while matching tmpfs on small-file reads.
Metadata stat operations are 12\% faster than ext4-dax.  \daxfs is the
first filesystem to provide a shared, writable namespace across CXL hosts
with zero-copy access and no locks on the data path.

\end{abstract}

\section{Introduction}
\label{sec:intro}
%-------------------------------------------------------------------------------

Byte-addressable persistent memory~\cite{intel17:pmem,izraelevitz19:pmem}
and CXL-attached memory~\cite{cxl22:spec} have introduced a
fundamentally new tier in the memory hierarchy: storage-class media
directly accessible via CPU load and store instructions.  The Linux
DAX (Direct Access) subsystem~\cite{linuxdax} maps this memory into
user address spaces without copying through the page cache, offering
DRAM-class latency with the capacity and sharing properties of storage.

CXL~3.0 introduces \emph{Global Fabric Attached Memory} (GFAM), where a
memory device connects to a CXL switch fabric and is accessible by
multiple hosts through hardware-coherent load/store operations.  For the
first time, independent servers can perform atomic read-modify-write
operations on shared memory with cache-line granularity and
hardware-enforced coherence, without a network stack, RDMA verbs, or software
coherence protocol.  A 64-bit \texttt{cmpxchg} issued by Host~A on a CXL
memory address is atomic with respect to a concurrent \texttt{cmpxchg}
by Host~B on the same address.

A natural use for CXL shared memory is a \emph{shared filesystem}:
multiple hosts reading and writing a common namespace without copying
data between per-host page caches or coordinating through a network
protocol.  Several important workloads demand exactly this capability:

\begin{itemize}
\item \textbf{LLM inference serving.}  Multiple GPU hosts share model
  weights through a common memory pool.  One copy in CXL memory serves
  all inference instances, eliminating the minutes-long cold-start
  latency of loading gigabyte-scale models per host.

\item \textbf{CXL memory pooling.}  CXL~3.0 enables memory pooling
  across hosts and devices.  A shared CXL memory pool holds common
  data (foundation model weights, embedding tables, reference
  datasets) accessed concurrently by multiple hosts or GPU tenants.
  Duplicating the shared data per host wastes scarce CXL capacity.

\item \textbf{Container rootfs sharing.}  $N$ containers running the same
  base image today require $N$ copies in host-local page caches.  A shared
  filesystem on CXL memory reduces this to a single copy.

\item \textbf{Multikernel shared rootfs.}  Independent kernel instances
  on shared hardware require a common root filesystem without per-kernel
  duplication or distributed locking.
\end{itemize}

Existing filesystems fail to serve this environment.
Table~\ref{tab:feature-matrix} summarizes the limitations.  Per-host
filesystems (tmpfs, ext4-dax, NOVA) maintain per-host metadata and page
caches, so $N$ hosts accessing the same data allocate $N$ copies.
Distributed filesystems (NFS, CephFS, Lustre) interpose a network
protocol between applications and data, adding latency and complexity
that CXL shared memory should eliminate.  Overlay-based solutions
(overlayfs, erofs+fscache) cannot coordinate across hosts and perform
copy-up operations that negate the sharing benefit.
FamFS~\cite{groves24:Famfs} targets CXL shared memory but uses a
single-master log-replay model that precludes concurrent writes from
client hosts, provides no shared caching, and lacks file mutation
operations (append, truncate, delete) on clients.

We present \daxfs, a Linux kernel filesystem module designed from the
ground up for CXL shared memory.  Rather than adapting block-device
abstractions, \daxfs treats byte-addressable memory as the storage
medium and introduces three key innovations:

\begin{enumerate}
\item \textbf{CAS-based hash overlay.}  All writes go through an
  open-addressing hash table where bucket insertions use a single
  64-bit \texttt{cmpxchg}.  No locks, no log, no journal.  Multiple
  hosts insert concurrently with linearizable semantics guaranteed by
  the CXL coherence protocol.

\item \textbf{Cooperative shared page cache.}  A demand-paged cache
  in DAX memory serves all hosts simultaneously.  Cache slot state
  transitions use \texttt{cmpxchg}, so exactly one host fills each
  slot while others wait.  Because the cache resides in CXL shared
  memory, filled slots are immediately visible to all hosts with no
  invalidation protocol.

\item \textbf{Flat validated format.}  The base image uses fixed-size
  inodes and flat directory entry arrays with no linked lists, no
  B-trees, and no pointer indirection.  This enables complete
  mount-time validation in a single bounded pass.
\end{enumerate}

\section{Motivation}
\label{sec:motivation}
%-------------------------------------------------------------------------------

\daxfs targets scenarios where byte-addressable shared memory is
available and traditional filesystem abstractions impose unnecessary
overhead.  We identify three fundamental problems that collectively
require a new filesystem design.

\paragraph{Per-Host Duplication.}
tmpfs, ext4-dax, and NOVA each maintain per-host metadata (superblock,
inode cache, dentry cache) and per-host page caches.  Even if two hosts
map the same physical memory, each host instantiates independent VFS
state.  When 8 hosts serve the same LLM model, this means 8 copies of
the page cache metadata and 8 independent directory caches, a pure
waste of CXL memory capacity.  A single 70B-parameter model at FP16
occupies 140\,GB; a pool serving $N$ tenants cannot afford $N$ copies.

\paragraph{No Multi-Host Coordination.}
Existing DAX filesystems assume a single-host, single-writer model.
ext4-dax uses journaling with per-host transaction IDs; NOVA uses
per-CPU log structures.  Neither can handle concurrent writers on
different hosts accessing the same metadata through shared memory.
FamFS~\cite{groves24:Famfs} is the closest existing system: it
targets CXL shared memory and supports multiple hosts mounting the
same DAX region.  However, FamFS uses a single-master log-replay
model where one designated host pre-allocates files and clients
replay its metadata log.  Clients cannot create, append, truncate,
or delete files, and there is no shared cache or CXL-atomic
coordination.  We provide a detailed architectural comparison in
\S\ref{sec:related}.

\paragraph{Network Protocol Overhead.}
Distributed filesystems (NFS, CephFS, Lustre) could serve
multi-host access, but they interpose a network protocol between
applications and byte-addressable memory, adding $\mu$s-scale
latency to what should be ns-scale load/store operations.
When the memory is already directly accessible via CXL, a network
filesystem negates the entire benefit of disaggregated memory.

These observations lead to four design requirements:
\begin{enumerate}
\item \textbf{Lock-free multi-host writes.}  Multiple hosts must
  create files, write data, and modify metadata concurrently without
  a central coordinator or distributed lock manager.
\item \textbf{Shared caching.}  A single cache instance in shared
  memory must serve all hosts, with coherent state transitions that
  prevent duplicate fills.
\item \textbf{Zero-copy access.}  Applications should \texttt{mmap}
  file data directly from CXL memory without copying through a
  per-host page cache.
\item \textbf{Bounded validation.}  Images from untrusted sources
  must be validatable in $O(n)$ time with no risk of infinite loops
  from malformed pointers.
\end{enumerate}

\section{Design}
\label{sec:design}
%-------------------------------------------------------------------------------

\daxfs is designed around three principles: (1)~\emph{memory is the
storage}, not a cache for a block device; (2)~\emph{reads should be
free}, resolving to direct pointer dereferences; and
(3)~\emph{coordination uses only hardware atomics}, with no locks on
the data path.

\subsection{Architecture Overview}

\daxfs operates on a contiguous DAX-mapped memory region containing
up to four areas laid out sequentially:

\subsection{Zero-Copy Read Path}
\label{sec:zero-copy}

In \daxfs, file reads resolve to direct pointers into DAX memory
without any intermediate copying.  The resolution path is:

\begin{enumerate}
\item Look up the data page in the hash overlay for key
  \texttt{(ino $\ll$ 20) | pgoff}.  If found, return a pointer to
  the overlay pool entry's data.
\item If absent, look up the base image's data area using the inode's
  stored data offset.
\item If the base image uses split mode with a backing file, look up
  the shared page cache for the page.
\end{enumerate}

At no point is data copied.  For \texttt{mmap}, the filesystem
installs PFN (page frame number) mappings directly to the DAX memory,
so user-space loads resolve to the physical memory in a single page
table walk.

\subsection{Hash Overlay: Lock-Free Writes via CAS}
\label{sec:overlay}

The hash overlay is the core mechanism that enables concurrent
multi-host writes.  It is an open-addressing hash table with linear
probing, stored entirely in DAX memory, where all mutations use a
single 64-bit \texttt{cmpxchg}.

\subsubsection{Bucket Structure}

Each bucket is 16 bytes:

\begin{lstlisting}
struct daxfs_ovl_bucket {
  uint64_t state_key; /* bit 0: USED/FREE
                         bits 63:1: key    */
  uint64_t pool_off;  /* offset into pool  */
};
\end{lstlisting}

Bit~0 of \texttt{state\_key} distinguishes FREE (0) from USED (1).
The remaining 63 bits encode the lookup key.

\subsubsection{Key Encoding}

\daxfs encodes four entry types into the 63-bit key space:

\begin{itemize}
\item \textbf{Data page}: \texttt{(ino $\ll$ 20) | pgoff}.  Supports
  up to $2^{20}$ pages (4\,GB) per file.
\item \textbf{Inode metadata}: \texttt{(ino $\ll$ 20) | 0xFFFFF}.
  The sentinel page offset distinguishes inode entries from data.
\item \textbf{Directory list head}: \texttt{(ino $\ll$ 20) | 0xFFFFE}.
  Points to the first overlay dirent for a directory.
\item \textbf{Directory entry}: \texttt{FNV-1a(parent\_ino, name)}.
  A 63-bit hash of the parent inode number and entry name.
\end{itemize}

\subsubsection{Insert Protocol}

To insert a new entry, a host:
\begin{enumerate}
\item Computes \texttt{hash = key \% bucket\_count}.
\item Reads \texttt{bucket[hash].state\_key}.
\item If FREE: attempts \texttt{cmpxchg(\&bucket[hash].state\_key, 0,
  key $\ll$ 1 | 1)}.  On success, allocates a pool entry and writes
  \texttt{pool\_off} with \texttt{smp\_wmb} ordering.  On failure
  (another host won the race), retries from step~2.
\item If USED with matching key: the entry already exists (update or
  conflict).
\item If USED with different key: linear probe to \texttt{hash+1} and
  repeat.
\end{enumerate}

This protocol is lock-free: no host can block another.  Under hash
collision, it degrades gracefully with bounded probing.

\subsubsection{Pool Allocator}

Pool entries are variable-size: inodes (32\,B), data pages
(4\,KB + 8\,B header), and dirents ($\sim$280\,B).  Allocation uses an
atomic bump pointer:

\begin{lstlisting}
off = atomic_fetch_add(&hdr->pool_alloc,
                       size);
\end{lstlisting}

This is a single atomic operation, contention-free across hosts.
Freed entries are recycled through per-type lock-free free lists (CAS
on list head), avoiding pool exhaustion for long-running workloads.

\subsubsection{Directory Operations}

Each directory maintains a per-directory linked list of overlay
dirents.  The list head is stored at key
\texttt{(dir\_ino $\ll$ 20) | 0xFFFFE}.  \texttt{readdir} iterates
both the base image dirent array and the overlay list, with overlay
entries taking precedence.  Deletion uses tombstone entries.

\subsection{Shared Page Cache}
\label{sec:pcache}

The shared page cache (\emph{pcache}) provides cooperative
demand-paged caching for backing store mode.  It resides in DAX
memory and is therefore directly accessible by all hosts sharing the
CXL memory region.

\subsubsection{Slot State Machine}

Each cache slot has a 16-byte metadata entry:

\begin{lstlisting}
struct daxfs_pcache_slot {
  __le64 state_tag; /* bits[1:0]=state,
     bits[5:2]=refcount, bits[63:6]=tag */
  __le32 ref_bit;   /* clock: recently used */
  __le32 reserved;
};
\end{lstlisting}

The state machine uses three states encoded in the low 2 bits:

\begin{center}
\texttt{FREE (0) $\xrightarrow{\texttt{cmpxchg}}$ PENDING (1)
  $\xrightarrow{\texttt{cmpxchg}}$ VALID (2)}
\end{center}

Bits [5:2] hold a 4-bit refcount (0--15 concurrent readers),
and the upper 58 bits encode the tag:
\texttt{(ino $\ll$ 20) | pgoff}, identifying which file page this
slot caches.  Packing state, refcount, and tag into a single 64-bit
word allows a single \texttt{cmpxchg} to atomically update all three
fields, which is essential for lock-free multi-host coordination.

\textbf{Fill protocol.}  When a host needs a page not in the cache:
\begin{enumerate}
\item Probe up to 8 slots starting at the hash position.  If a
  matching VALID slot is found, pin it (CAS-increment refcount) and
  return.  If a FREE slot is found, record it.
\item \texttt{cmpxchg} the FREE slot from FREE to PENDING with the
  desired tag.  If another host wins, retry.
\item Read the backing file page into the slot's data area via
  \texttt{kernel\_read}.
\item \texttt{cmpxchg} the slot from PENDING to VALID.
\end{enumerate}

Other hosts that need the same page during step~3 see the PENDING
state and busy-poll until VALID, avoiding duplicate reads from the
backing file.

\subsubsection{MH-Clock Eviction}

When all probe slots are occupied, \daxfs uses a multi-host clock
(MH-clock) eviction algorithm that combines local probe-window
eviction with a global background sweep.  Unlike a standard clock
algorithm with a single centralized hand, MH-clock is fully
decentralized: each host independently selects victims within its
local probe window using three escalating phases:

\begin{enumerate}
\item \textbf{Cold victim.}  Scan the 8-slot probe window for a
  VALID slot with \texttt{ref\_bit}=0 and refcount=0.  If found,
  CAS it to FREE and restart the fill.
\item \textbf{Clear and yield.}  If all probed slots are hot
  (\texttt{ref\_bit}=1), clear their \texttt{ref\_bit} fields and
  yield the CPU briefly (\texttt{cpu\_relax}).  This gives other
  hosts an opportunity to re-touch genuinely hot entries before
  the re-scan.
\item \textbf{Re-scan.}  Scan again for a cold, unpinned victim.
  If still none, force-evict the first VALID slot with refcount=0,
  ignoring \texttt{ref\_bit}.
\end{enumerate}

A separate background clock sweep periodically advances a shared
atomic \texttt{evict\_hand} counter by 64 slots using
\texttt{cmpxchg}; the host that wins the CAS clears
\texttt{ref\_bit} on all VALID slots in that window.  Hosts that
lose the race skip the sweep, ensuring that only one host clears
each window and ref\_bits decay at a controlled rate even under
contention.

The refcount in \texttt{state\_tag} ensures that slots actively
being read by any host are never evicted.  Readers CAS-increment
the refcount before accessing slot data and CAS-decrement it
afterward, providing safe pinning without locks.

\subsubsection{Multi-File Support}

The tag encoding \texttt{(ino $\ll$ 20) | pgoff} supports up to
$2^{20}$ pages (4\,GB) per file.  Multiple backing files share the
same cache through a backing file array in the superblock, with inode
numbers namespaced per file.

\subsection{Cross-Host Coherency}
\label{sec:coherency}

Because \daxfs uses \texttt{cmpxchg} on CXL-coherent memory for all
state transitions, cross-host coherency is provided by hardware.
Two specific mechanisms warrant discussion:

\paragraph{i\_size coherency.}
When Host~A appends data and updates the file size in the overlay
inode entry, Host~B must see the new size.  \daxfs stores the
authoritative \texttt{i\_size} in the overlay inode entry (DAX memory).
On each read path entry, \daxfs performs a \texttt{READ\_ONCE} on
the overlay's size field and updates the in-kernel VFS inode,
ensuring reads always see the latest file size without explicit
invalidation messages.

\paragraph{Memory ordering.}
Insert operations use \texttt{smp\_wmb} between writing pool entry
contents and writing the bucket's \texttt{pool\_off} field.
Lookup operations use \texttt{smp\_rmb} after reading
\texttt{pool\_off} to ensure they see the complete pool entry.
On x86, these compile to compiler barriers (x86 provides TSO
ordering); on ARM64, they emit fence instructions.

\subsection{On-Disk Format}
\label{sec:format}

\daxfs uses a deliberately simple flat format (version~7) designed for
safe handling of untrusted images and efficient direct access.

\subsubsection{Region Layout}

The memory region is divided into up to four areas, laid out
sequentially:

\begin{description}
\item[Superblock (4\,KB).]  Magic number (\texttt{0x64617835},
  ``dax5''), version, flags, total size, and byte offsets to all
  other regions.

\item[Base image (optional).]  A self-contained read-only filesystem
  snapshot with a 4\,KB header, fixed-size inode table (64 bytes per
  inode), and data area containing file contents and flat directory
  entry arrays.

\item[Hash overlay (optional).]  A 4\,KB header followed by the
  bucket array (\texttt{bucket\_count $\times$ 16} bytes) and a
  variable-size pool for overlay entries.  The header stores magic,
  bucket count, pool offsets, and atomic counters.

\item[Page cache (optional).]  A 4\,KB header followed by slot
  metadata (\texttt{slot\_count $\times$ 16} bytes) and slot data
  (\texttt{slot\_count $\times$ 4\,KB} pages).
\end{description}

\subsubsection{Flat Directory Format}

Directories store fixed-size entries of 271 bytes each (the
\texttt{daxfs\_dirent} structure), containing a name length, an inode
number, a parent inode number, and a 254-byte inline name.  This flat
design offers several security and performance advantages over
pointer-based directory formats (\eg htree in ext4, B-tree in Btrfs,
linked lists in tmpfs):

\begin{itemize}
\item \textbf{No pointer chasing.}  Directory entries are stored as
  flat arrays.  There are no linked lists, B-trees, or hash tables
  that could contain cycles or dangling pointers.

\item \textbf{Bounded iteration.}  The fixed entry size means
  directory traversal is a simple loop with a known upper bound.

\item \textbf{Inline names.}  File names are stored directly in the
  entry, eliminating string table indirection and out-of-bounds
  vulnerabilities.

\item \textbf{Trivial validation.}  The entire format can be validated
  in a single pass at mount time without complex tree traversal.
\end{itemize}

\subsubsection{Overlay Pool Entries}

Pool entries are typed and variable-size:

\begin{itemize}
\item \textbf{Inode entry (32\,B):}  Mode, uid, gid, size, timestamps.
\item \textbf{Data entry (4\,KB + 8\,B):}  A type header followed by a
  4\,KB page of file data (COW granularity).
\item \textbf{Dirent entry ($\sim$280\,B):}  Parent inode, name, target
  inode, mode, and a tombstone flag for deletions.
\item \textbf{Dirlist entry (16\,B):}  Per-directory list head linking
  overlay dirents.
\end{itemize}

Each entry type has a per-type free list for recycling.  The free list
head is stored in the overlay header and updated via \texttt{cmpxchg},
making recycling lock-free across hosts.

%-------------------------------------------------------------------------------
\section{Implementation}
\label{sec:impl}
%-------------------------------------------------------------------------------

\daxfs is implemented as a loadable Linux kernel module in
approximately 2,500 lines of C.  It registers a filesystem type
(\texttt{daxfs}) and supports both the legacy \texttt{mount(2)}
interface and the modern \texttt{fsopen}/\texttt{fsconfig}/\texttt{fsmount}
interface introduced in Linux~5.6.

\subsection{Memory Mapping}

\daxfs supports two memory backing modes:
\begin{itemize}
\item \textbf{Physical address.}  The \texttt{phys=} and \texttt{size=}
  mount options cause \daxfs to call \texttt{memremap()} on the
  specified physical range.  This is the primary path for Optane PMem
  and CXL memory devices.
\item \textbf{DMA buffer.}  Using the \texttt{fsopen} API, a
  user-space process passes a dma-buf file descriptor via
  \texttt{FSCONFIG\_SET\_FD}.  \daxfs calls \texttt{dma\_buf\_vmap()}
  to obtain a kernel virtual mapping.  This enables GPU/accelerator
  memory integration.
\end{itemize}

Both modes bypass the kernel's \texttt{dax\_device} abstraction entirely,
mapping memory directly into the filesystem's address space.

\subsection{VFS Integration}

\daxfs registers standard VFS operations:

\begin{itemize}
\item \textbf{Inode operations}: lookup, create, mkdir, symlink,
  rename, unlink, setattr.  All write operations go through the hash
  overlay; the base image is never modified.
\item \textbf{File operations}: \texttt{read\_iter} performs zero-copy
  reads by returning pointers into DAX memory.  For pcache-backed
  data, the read path pins the cache slot (incrementing refcount)
  before returning data, preventing eviction during the read.
\item \textbf{Address space operations}: \texttt{readpage}/\texttt{readahead}
  for integration with the kernel page cache on non-DAX access paths.
\end{itemize}

Data resolution on every read path calls
\texttt{daxfs\_refresh\_isize()}, which performs a \texttt{READ\_ONCE}
on the overlay's size field, ensuring cross-host coherency of file
sizes without explicit invalidation.

\subsection{Kernel Compatibility}

\daxfs supports Linux 5.4 through 6.x via a centralized compatibility
layer (\texttt{daxfs.h}) that provides shims for API changes across 12
kernel version boundaries, including the \texttt{user\_namespace}
$\to$ \texttt{mnt\_idmap} transition (5.12/6.0), the \texttt{fs\_context}
mount API (5.6), and timestamp accessor changes (6.6).

\subsection{Userspace Tools}

\textbf{mkdaxfs} creates filesystem images from directory trees.  It
supports static, split, and empty modes, with configurable overlay
pool size (default 64\,MB), hash table bucket count (default 65,536),
and page cache slot count.  It can allocate from DMA heaps and
optionally mount the image after creation.

\textbf{daxfs-inspect} provides read-only inspection of mounted
instances, reporting memory layout, overlay hash table utilization
(bucket occupancy, entry type distribution), and pool usage statistics.

\subsection{Security and Validation}

The flat format enables complete validation in a single sequential
scan.  When the \texttt{validate} mount option is specified, \daxfs
checks:
\begin{itemize}
\item Superblock magic and version.
\item All inode table entries: bounds on data offsets and sizes.
\item All directory entries: name lengths, inode references.
\item Region overlap detection.
\end{itemize}

The absence of pointer-based structures eliminates an entire class of
attacks: cycle injection, dangling pointer exploitation, and unbounded
traversal, which is important for mounting untrusted images in container
environments.

%-------------------------------------------------------------------------------
\section{Evaluation}
\label{sec:evaluation}
%-------------------------------------------------------------------------------

We evaluate \daxfs against three baselines on a single host to measure
the raw performance of \daxfs's sharing-ready design.  Multi-host
evaluation on CXL hardware is discussed in \S\ref{sec:multihost}.

\subsection{Experimental Setup}

\textbf{Hardware.}  Intel Xeon Gold 5418Y (48 cores, 96 threads),
128\,GB DDR4, Intel Optane Persistent Memory 200 Series (pmem0).
The Optane device provides byte-addressable DAX memory via
\texttt{memremap}, exercising the same code path that CXL memory
devices use.

\textbf{Software.}  Linux 5.1.0+, Ubuntu.  All filesystems are
configured for DAX mode where supported.

\textbf{Baselines.}
\begin{itemize}
\item \textbf{ext4-dax}: ext4 mounted with \texttt{-o dax} on a pmem
  device.  Represents the mature DAX filesystem baseline.
\item \textbf{NOVA}~\cite{xu16:nova}: Log-structured filesystem for
  persistent memory.  Represents the state-of-the-art PM-optimized
  filesystem.
\item \textbf{tmpfs}: In-DRAM filesystem.  Represents the upper bound
  for memory-speed access (no persistence, no DAX, pure DRAM).
\end{itemize}

\textbf{Workload.}  We populate each filesystem with 1,000 small files
(4\,KB each) and several large files (1\,MB, 16\,MB).  Sequential reads
use \texttt{dd} with varying block sizes.  Caches are dropped before
each measurement.  Results are averaged over 5 runs.

\subsection{Sequential Read Throughput}

% \begin{figure}[t]
% \centering
% \includegraphics[width=\columnwidth]{bench/results/read_throughput.png}
% \caption{Sequential read throughput comparison.  Left: 16\,MB file
%   (1\,MB block size).  Right: 1\,MB file (4\,KB block size).
%   \daxfs achieves $4.1\times$ higher throughput than ext4-dax on
%   large files and matches tmpfs on small files.}
% \label{fig:read_throughput}
% \end{figure}

Figure~\ref{fig:read_throughput} shows sequential read throughput for
two file sizes.

\textbf{Large file (16\,MB).}  \daxfs achieves 104.7\,MB/s,
$4.1\times$ higher than ext4-dax (25.8\,MB/s) and $22.3\times$ higher
than NOVA (4.8\,MB/s).  tmpfs achieves 228.6\,MB/s, roughly $2\times$
\daxfs, reflecting the DRAM vs.\ Optane PMem bandwidth gap.
The \daxfs advantage over ext4-dax comes from eliminating the block
layer: \daxfs resolves a file offset to a DAX memory pointer in a
single hash lookup (or direct offset computation for base image data),
while ext4-dax traverses extent trees, acquires inode locks, and
performs iomap callbacks.

\textbf{Small file (1\,MB).}  \daxfs achieves 100.0\,MB/s, matching
tmpfs exactly and $21\times$ faster than both ext4-dax and NOVA
(4.8\,MB/s each).  For small files, \daxfs's flat inode table and
inline data layout provide direct memory access with negligible
metadata overhead, achieving DRAM-equivalent throughput even on PMem.

\subsection{Read Latency}

% \centering
% \includegraphics[width=\columnwidth]{bench/results/read_latency.png}
% \caption{Sequential read latency (lower is better).  \daxfs achieves

%   $15\times$ lower than NOVA.}
% \label{fig:read_latency}
% \end{figure}

Figure~\ref{fig:read_latency} shows read latency for the same
workloads.  \daxfs completes a 16\,MB sequential read in 153.6\,ms
versus ext4-dax's 623.0\,ms ($4.1\times$ slower) and NOVA's
3,347.0\,ms ($21.8\times$ slower).  For the 1\,MB file, \daxfs and
tmpfs both complete in 14.0\,ms, while ext4-dax and NOVA require
$\sim$212\,ms ($15\times$ slower).

The NOVA result is notably poor because NOVA's per-inode log structure
requires traversing log entries to locate data extents, and its radix
tree index adds overhead that dominates for sequential scans.  ext4-dax
performs better than NOVA but still incurs extent tree traversal and
iomap overhead on every \texttt{read\_iter} call.

\subsection{Metadata Operations}

% \begin{figure}[t]
% \centering
% \includegraphics[width=\columnwidth]{bench/results/metadata.png}

%   Left: \texttt{readdir}.  Right: \texttt{stat}.}

% \end{figure}

Figure~\ref{fig:metadata} shows metadata operation latency on a
directory containing 1,000 files.

\textbf{stat (1,000 files).}  \daxfs completes in 3,550\,ms, 12\%
faster than ext4-dax (4,029\,ms), 5\% faster than NOVA (3,748\,ms),
and 11\% faster than tmpfs (3,974\,ms).  The \daxfs advantage comes
from its flat inode table: \texttt{stat} resolves to a single indexed
read into a contiguous array, with no inode cache miss penalty and no
extent tree traversal.

\textbf{readdir (1,000 files).}  \daxfs completes in 40.0\,ms,
comparable to tmpfs (39.8\,ms) and NOVA (41.9\,ms).  ext4-dax is
fastest at 23.6\,ms, $1.7\times$ faster than \daxfs, due to its
HTree directory index.  \daxfs's flat dirent array requires a linear
scan, which is competitive for moderate directory sizes but does not
benefit from indexing.

\subsection{Multi-Host Evaluation}
\label{sec:multihost}

Multi-host CXL~3.0 evaluation is pending hardware availability.
All \daxfs coordination primitives (\texttt{cmpxchg} on overlay
buckets, page cache slots, and pool free lists) are designed for
cross-host atomicity and have been validated in single-host
concurrent tests.  The Optane PMem evaluation above exercises the
same \texttt{memremap}-based code path that CXL memory devices use,
so we expect the single-host performance characteristics to carry
over.

We plan to evaluate on a dual-host CXL~3.0 switch fabric with shared
memory, measuring: (1)~concurrent write throughput scaling with host
count, (2)~shared page cache hit rates under multi-host access
patterns, (3)~overlay hash table contention under concurrent inserts
from multiple hosts, and (4)~end-to-end LLM inference cold-start
latency with shared model weights versus per-host loading.

%-------------------------------------------------------------------------------
\section{Related Work}
\label{sec:related}
%-------------------------------------------------------------------------------

\subsection{Persistent Memory Filesystems}

PMFS~\cite{dulloor14:pmfs} was the first filesystem designed for
persistent memory, using a B-tree for file indexing and copy-on-write
for consistency.  NOVA~\cite{xu16:nova} introduced per-inode log
structuring, achieving high concurrency by eliminating the global
journal.  Strata~\cite{kwon17:strata} splits the I/O path between
user-space and kernel layers, while
SplitFS~\cite{kadekodi19:splitfs} intercepts POSIX calls to reduce
kernel crossings.

All of these are single-host filesystems.  They maintain complex
metadata structures (B-trees, per-inode logs, radix trees) that
cannot be shared across hosts without a distributed locking protocol.
\daxfs eliminates these structures in favor of a flat base image
overlaid by a lock-free hash table, enabling multi-host coordination
through hardware atomics alone.

\subsection{CXL Shared Memory Systems}

Pond~\cite{li23:pond} explores CXL memory pooling for cloud
applications.  TPP~\cite{maruf23:tpp} provides transparent page
placement across DRAM and CXL memory tiers.  These systems focus on
memory management rather than filesystem semantics.

FamFS~\cite{groves24:Famfs} is architecturally the closest prior work
to \daxfs.  Both provide a filesystem interface to CXL shared memory,
supporting multiple hosts mounting the same DAX region.  However, the
two systems make fundamentally different design choices, summarized in
Table~\ref{tab:famfs-comparison}.

\begin{table}[t]
\centering
\footnotesize
\begin{tabular}{@{}lll@{}}
\toprule
 & \textbf{\daxfs} & \textbf{FamFS} \\
\midrule
Coordination    & Peer-to-peer \texttt{cmpxchg} & Single master log \\
Writes          & Any host, lock-free CAS        & Master only \\
Shared cache    & Cooperative pcache             & None \\
Allocation      & Internal bump allocator        & Master-managed extents \\
File ops        & Create/read/write/delete       & Pre-allocate only \\
Image model     & Self-contained layered image   & Individual mapped extents \\
Layered storage & Base image + COW overlay       & None \\
CXL atomics     & Core design primitive          & Not used \\
\bottomrule
\end{tabular}
\caption{Architectural comparison of \daxfs and FamFS.}
\label{tab:famfs-comparison}
\end{table}

FamFS is a thin mapping layer: a single master host pre-allocates
files as individually mapped extents and writes a metadata log that
client hosts replay.  Clients cannot create files, append data,
truncate, or delete; writable access requires users to manage
coherency externally.  FamFS does not use CXL atomics for
coordination, has no shared cache, no layered storage, and no
self-contained image format.  \daxfs addresses all of these gaps
through its CAS-based overlay, cooperative page cache, and layered
image model, enabling fully concurrent multi-host access without a
central coordinator.

\subsection{Distributed and Network Filesystems}

GFS~\cite{ghemawat03:gfs}, CephFS~\cite{weil06:ceph}, and
Lustre~\cite{schwan03:lustre} provide shared filesystems over
networks.  Orion~\cite{yang20:orion} targets RDMA-attached persistent
memory.  Assise~\cite{anderson20:assise} uses client-local NVM for a
distributed filesystem.  These systems use message-passing or RPC for
coordination, adding $\mu$s-scale latency that CXL shared memory
makes unnecessary.  \daxfs replaces network coordination with
hardware atomics on shared memory.

\subsection{Read-Only and Container Filesystems}

EROFS~\cite{gao20:erofs} provides compressed read-only images with
fscache-based demand paging.  Nydus and Stargz provide lazy container
image pulling.  composefs shares file content across images via
content-addressed storage.  These focus on image distribution
efficiency; none support multi-host runtime sharing or concurrent
writes.

\subsection{Lock-Free Data Structures}

\daxfs's overlay draws on classic lock-free hash table
designs~\cite{michael02:lockfree,herlihy08:art}, adapted for DAX
memory where persistence ordering matters and cache-line granularity
affects contention.  The bump allocator with free-list recycling
follows epoch-based reclamation patterns~\cite{michael04:hazard}.

%-------------------------------------------------------------------------------
\section{Discussion and Limitations}
\label{sec:discussion}
%-------------------------------------------------------------------------------

\paragraph{Flat directory scalability.}
\daxfs's flat directory format performs a linear scan for lookup,
which is $O(n)$ in directory size.  For directories with more than
$\sim$10K entries, an overlay-indexed lookup would improve performance.
For moderate directories (1K entries), the linear scan is competitive
with indexed approaches (Figure~\ref{fig:metadata}).

\paragraph{Fixed overlay sizing.}
The overlay hash table size is fixed at creation time.  Dynamic
resizing would require a stop-the-world migration across all hosts.
We recommend sizing the bucket count to maintain $<$70\% load factor:
at 75\% load, average probe length is 2.5; at 90\%, it is 5.5.

\paragraph{Persistence guarantees.}
\daxfs currently relies on ADR (Asynchronous DRAM Refresh) or eADR
for persistence.  It does not issue explicit
\texttt{clflush}/\texttt{clwb} instructions.  On platforms without
ADR, a crash could lose recent overlay insertions.  Adding explicit
cache-line flushes is straightforward but adds latency.

\paragraph{Pool recycling.}
The overlay pool uses per-type lock-free free lists for recycling
deleted entries, but the pool itself is not compacted.  Long-running
workloads with heavy churn may fragment the pool.  Online compaction
is possible future work.

\paragraph{POSIX compliance gaps.}
\daxfs does not support \texttt{mknod} (device nodes, FIFOs, sockets)
or extended attributes.  File names are limited to 255 characters.
These restrictions reflect the target workloads (container rootfs,
shared caching) where these features are rarely needed.

%-------------------------------------------------------------------------------
\section{Conclusion}
\label{sec:conclusion}
%-------------------------------------------------------------------------------

CXL shared memory enables a new filesystem design point: multiple
hosts sharing a single namespace and cache through hardware-coherent
load/store operations, with no network protocol overhead.  \daxfs
exploits this by using \texttt{cmpxchg} as its sole coordination
primitive, achieving lock-free concurrent writes, cooperative caching,
and zero-copy access in a unified design.

Our evaluation shows that \daxfs achieves $4.1\times$ the sequential
read throughput of ext4-dax, $22\times$ that of NOVA, and matches
tmpfs on small-file reads, while providing multi-host sharing
semantics that none of these baselines support.  As CXL~3.0 hardware
becomes available, \daxfs provides a ready filesystem layer for
disaggregated memory pools.
